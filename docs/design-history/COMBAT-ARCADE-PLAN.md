# Аркадная боёвка в режиме исследования — план интеграции

Модель: **tab-target + виртуальный джойстик** (мобильная схема, вид сверху, реальное время).
Принцип: аркадной становится **только боёвка встреч в полёте** (`fromFlight`). Пошаговый бой в `08-battle.js` остаётся для «Дружеского турнира» и «Испытания волн» — его не трогаем. Так переделка изолирована и легко откатывается.

---

## 1. Что есть сейчас (реальные швы в коде)

**Вход в бой** — `js/07-flight.js`, функция `encounter(kind, ent)` → обработчик `#fcvFight` (строки ~298–312):

```
f._pend = {kind, ent}; f.battleWin = undefined;   // запомнили встречу
cancelAnimationFrame(f.raf); f.raf = 0;            // остановили полёт
fs.style.display='none';                           // спрятали #flightFs
switchView('arena'); startBattle(f.d, {...});      // ушли в пошаговый бой
battle.fromFlight = true;
```

**Выход из боя** — `js/08-battle.js`, `endBattle()` (строки ~665–681): при `fromFlight` ставит `flight.battleWin = win`, рисует кнопку «↩ Вернуться в полёт», по клику прячет `#battleStage`, показывает `#flightFs` и зовёт `renderFlight()`.

**Применение итога** — `js/07-flight.js`, `renderFlight()` (строки ~214–247): читает `f._pend` + `f.battleWin`. Победа → `addEgg()`, монеты вокруг логова, `f.beasts++`. Поражение → дракон восстаёт с 25% HP. Затем `f.paused=false` и цикл кадров запускается заново.

**Дроп → сейв** — прямые мутации `S.gold`, `S.dust`, `addEgg()` + `persist()` (localStorage через безопасную обёртку в `11-save-init.js`). Синхронизация уже двусторонняя: полёт пишет в общий `S`, а `persist()` сохраняет.

**Вывод:** контракт «полёт → бой → возврат с результатом `win`» уже существует и работает. Нам достаточно **подменить середину** — то, что происходит между «начали бой» и «узнали `win`». Всё остальное переиспользуется без единой правки.

## 2. Стратегия: сохранить контракт, заменить движок

Новый аркадный бой встаёт **на тот же контракт**, что и пошаговый:

| Шаг | Пошаговый (сейчас) | Аркадный (станет) |
|---|---|---|
| Запуск | `startBattle(f.d, foe)` + `switchView('arena')` | `startArcadeFight(f.d, ent, {reward})` — свой fullscreen-оверлей |
| Процесс | ходы, мана, `#battleStage` | реальный цикл: джойстик, автоатака, кулдауны, AI |
| Итог | `flight.battleWin = win` | `flight.battleWin = win` (то же самое) |
| Возврат | `renderFlight()` | `renderFlight()` (то же самое) |

Поскольку выходной сигнал идентичен (`f._pend` + `f.battleWin`), **весь код применения награды и возврата в полёт остаётся как есть**. Риск переделки ограничен одним новым модулем и одним изменённым обработчиком.

## 4. Ядро аркадного боя (что кладём в 08b)

### 4.1 Переиспользуем боевые статы
`makeCombatant(d, isFoe)` из 08 уже даёт `maxHp/atk/def/spd/mana/spell/ult`. Из него сидируем аркадные сущности — балансировка остаётся согласованной с остальной игрой:

```js
const c = makeCombatant(myDragon, false);
const player = {
  x, y, hp:c.hp, maxHp:c.maxHp, speed: 120 + c.spd*4,
  mana:c.mana, manaMax:c.manaMax, manaRegen: 2,
  target:null, globalCd:0,
  autoAttack:{ cd: 1.2, remaining:0, range:150, dmg: c.atk },
  abilities: buildAbilities(c),   // из c.spell / c.ult
};
```

### 4.2 Перевод пошаговых величин в реальное время

| Пошаговое | Аркадное | Формула-ориентир |
|---|---|---|
| Урон за ход | Урон автоатаки за ~1.2 с | `dmg = atk` (та же величина, реже) |
| Стихийное преимущество (`ADVANTAGE`) | Множитель урона | `×1.4` контр / `×0.7` против |
| Заклинание (мана 3) | Способность с кулдауном 4–6 с + мана | из `c.spell.pow` |
| Ульта (мана 6) | Способность 12+ с, мощная AoE/heal | из `c.ult.pow` |
| Защита копит ману | ManaRegen со временем | `manaRegen`/сек |

### 4.3 Игровой цикл (свой RAF, не пересекается с полётным)
```js
function update(dt){
  player.x += joy.dx*player.speed*dt; player.y += joy.dy*player.speed*dt;
  player.mana = Math.min(player.manaMax, player.mana + player.manaRegen*dt);
  tickCooldowns(dt);            // globalCd, autoAttack, abilities
  autoAttackTarget(dt);         // бьёт выбранную цель в радиусе
  updateEnemyAI(dt);            // агро-радиус, преследование, телеграф+удар
  resolveDamage();              // хитбоксы AABB/окружности, i-frames
  if(enemyDead) return finish(true);
  if(player.hp<=0) return finish(false);
}
function finish(win){ cancelAnimationFrame(raf); hideOverlay();
  flight.battleWin = win; renderFlight(); }   // ← контракт возврата
```

### 4.4 Управление (tab-target + джойстик)
- **Джойстик слева** двигает дракона (переиспользовать логику `joy` из полёта: `onpointerdown/move/up` на элементе).
- **Тап по врагу** — `selectTarget(worldX, worldY)`: ближайший враг в радиусе тапа становится `player.target`, под ним рисуется кольцо, сверху — полоска HP. Тап по земле снимает цель.
- **Автоатака** бьёт цель сама, пока та в радиусе и `remaining<=0`.
- **Панель способностей справа** (2–3 кнопки): проверка `remaining<=0 && mana>=cost && цель в range`, глобальный кулдаун `globalCd=1c` против спама.

### 4.5 «MMO-вкус» (что берём из механик MMO)
Агро-радиус (враг активируется при входе в зону), **телеграф** удара врага (замах/красная зона за 0.5–1 с — успеть выйти), ресурс маны с регеном, GCD, i-frames после получения урона, фидбек (вспышка, тряска, частицы). Всё — чистый Canvas 2D top-down, как в полёте.

### 4.6 Дроп остаётся в существующем пути
Награду **не дублируем** в 08b. Аркадный бой лишь сообщает `win`, а начисление яйца/монет выполняет уже написанный блок в `renderFlight()` (строки ~219–235). Значит дроп (🥚 через `addEgg`, 🪙 монеты, `f.beasts++`) и синхронизация в сейв через `persist()` — без изменений и без риска рассинхрона.
