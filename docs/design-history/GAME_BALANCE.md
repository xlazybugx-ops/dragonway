# GAME_BALANCE.md — единый конфиг баланса

Весь баланс вынесен из игровой логики в один конфиг-объект **`GAME_BALANCE`** (короткий алиас **`GB`**),
файл **`js/00-balance.js`**, подключаемый **первым**. Формулы теперь читают коэффициенты из `GB`, а не из
«магических чисел» в коде. Значения — **ровно те же**, что были: баланс, логика, UI, анимации не изменены
(проверено сверкой значений и `node --check`).

Тюнинг игры теперь = правка одного файла `js/00-balance.js`.

---

## Архитектура

- `js/00-balance.js` — единственный источник кросс-системных коэффициентов баланса (`GAME_BALANCE`/`GB`).
- Формульные функции (`doStrike`, `advMult`, `xpToNext`, `rollEggRarity`, `forgeCost`, `smithyCost`,
  `ar_mitig`, `ar_elMul` и др.) берут числа из `GB`.
- **Данные-таблицы** (MOVES, EGG_RARITY, NATURES, CHEST_TYPES, статы видов `SPECIES`, множители боссов) —
  остаются собственными типизированными структурами: это правильная архитектура (параметры сущности живут
  рядом с сущностью), они уже отделены от логики и служат конфигом своих доменов.

## Рекомендации по дальнейшему развитию

1. **Дальнейшая централизация данных-таблиц**: при желании MOVES/EGG_RARITY/CHEST_TYPES можно тоже
   собрать под `GB` (напр. `GB.Skills`, `GB.Chests`), но полезнее держать их типизированными структурами.
2. **Профили баланса**: несколько наборов `GB` (например «казуальный»/«хардкор») — переключаемый пресет.
3. **Валидатор**: юнит-проверка, что все ключи `GB` присутствуют и в разумных диапазонах при загрузке.
4. **Горячий тюнинг**: dev-панель, меняющая `GB` в рантайме, для быстрой настройки без пересборки.
5. **Версионирование баланса**: поле `GB.version` для трекинга изменений между сборками.
